# 🔄 변경이력 관리 페이지 — 아키텍처 비교 분석

---

## 대상 변경이력 3종

| # | 변경이력 유형 | 대상 모듈 | 핵심 항목 |
|---|-------------|----------|----------|
| 1 | **담당자(상담매니저) 변경** | 준회원 | 회원명, 이전→변경 상담매니저, 분배일, 변경일, 처리자, 사유 |
| 2 | **매칭매니저 변경** | 정회원 | 회원명(성별), 회원상태, 변경전→변경후 매칭매니저, 변경일, 사유, 처리자 |
| 3 | **회원상태 변경** | 정회원 | (첨부이미지 참고) |

---

## 🅰️ 안 A: 탭 통합 방식 (1페이지 + N탭)

```
사이드바 메뉴: "변경이력 관리" (단일)
  ┗ /pages/management/change-history.html

  ┌───────────────────────────────────────────────────┐
  │  [담당자변경] [매칭매니저변경] [회원상태변경]  ← 탭  │
  ├───────────────────────────────────────────────────┤
  │  검색 필터 (탭별로 다른 필터)                      │
  │  ────────────────────────────────────────────      │
  │  결과 테이블 (탭별로 다른 컬럼)                    │
  └───────────────────────────────────────────────────┘
```

### ✅ 장점
| 항목 | 설명 |
|------|------|
| **업무 효율** | 한 화면에서 모든 변경이력 확인 가능. 탭 전환만으로 빠르게 비교 |
| **메뉴 간결** | 사이드바에 메뉴 1개만 추가 → 메뉴 비대화 방지 |
| **유지보수** | 공통 컴포넌트(검색, 테이블, 페이징) 재사용 극대화 |
| **기존 패턴과 일치** | 보유DB 현황 (`db-status.html?tab=db/perf`) 등 기존 탭 패턴과 동일 |
| **실무 레퍼런스** | 첨부이미지의 기존 시스템도 통합 방식 사용 중 |

### ⚠️ 단점
| 항목 | 설명 |
|------|------|
| **JS 파일 크기** | 탭별 로직이 하나의 파일에 집중되어 코드가 길어질 수 있음 |
| **필터 복잡도** | 탭마다 검색 필터가 다르므로 탭 전환 시 필터 초기화 로직 필요 |
| **권한 분리 한계** | 특정 탭만 권한 제한하려면 추가 로직 필요 |

## 🅱️ 안 B: 개별 분리 방식 (N페이지)

```
사이드바 메뉴:
  준회원 관리
    ┗ 담당자 변경이력          → /pages/associate/change-history.html
  정회원 관리
    ┗ 매칭매니저 변경이력       → /pages/regular/manager-change.html
    ┗ 회원상태 변경이력         → /pages/regular/status-change.html
```

### ✅ 장점
| 항목 | 설명 |
|------|------|
| **소속 명확** | 준회원/정회원 각 메뉴 하위에 위치 → 찾기 직관적 |
| **독립 배포** | 각 페이지가 독립적이라 한쪽 수정이 다른 쪽에 영향 없음 |
| **권한 제어 용이** | 페이지 단위 권한 체크 가능 (`canAccess('associate-change-history', role)`) |

### ⚠️ 단점
| 항목 | 설명 |
|------|------|
| **메뉴 비대화** | 사이드바에 메뉴 3개 추가 → 이미 많은 메뉴에 추가 부담 |
| **코드 중복** | 검색 필터, 테이블, 페이징 등 유사 UI를 3번 작성 |
| **교차 확인 불편** | 한 회원의 여러 변경이력을 보려면 페이지를 오가야 함 |

## 💡 추천: 안 A (탭 통합) + 하이브리드

> [!IMPORTANT]
> **안 A를 기본으로 하되**, 각 모듈 메뉴에서도 바로 진입할 수 있도록 **딥링크**를 지원하는 하이브리드 방식을 추천합니다.

### 구현 방식

```
파일 구조:
  pages/management/change-history.html    ← 메인 페이지
  pages/management/change-history.js      ← 메인 로직
  pages/management/ch-tab-consultant.js   ← 탭1: 담당자(상담매니저) 변경
  pages/management/ch-tab-manager.js      ← 탭2: 매칭매니저 변경
  pages/management/ch-tab-status.js       ← 탭3: 회원상태 변경

URL 딥링크:
  /pages/management/change-history.html              → 기본(탭1 활성)
  /pages/management/change-history.html?tab=consultant → 담당자 변경
  /pages/management/change-history.html?tab=manager    → 매칭매니저 변경
  /pages/management/change-history.html?tab=status     → 회원상태 변경
```

### 메뉴 배치

```diff
  사이드바 메뉴 (menu.js):

  정회원 관리
    ├── 정회원 목록
    ├── 매칭 대상자 검색
+   ├── 변경이력 관리         → change-history.html (탭 통합)
    ├── 약속/미팅 조회
    └── ...
```

### 추천 이유

| 근거 | 설명 |
|------|------|
| 1. 기존 패턴 | `db-status.html?tab=db/perf` 와 동일한 탭+쿼리 패턴 |
| 2. 실무 관행 | 첨부이미지의 레거시 시스템도 통합 관리 방식 |
| 3. 확장성 | 향후 "프로그램 변경", "지사 변경" 등 추가 시 탭만 추가하면 됨 |
| 4. 코드 효율 | 탭별 JS 분리로 파일 크기 관리 + 공통 로직 재사용 |
| 5. 사용자 경험 | 한 화면에서 종합적으로 변경이력 파악 가능 |
