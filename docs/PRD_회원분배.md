## 2. 페이지 구성

### 2.1 회원분배 (`distribute.html`)

| 항목 | 내용 |
|------|------|
| **파일** | `pages/management/distribute.html` + `distribute.js` |
| **페이지 ID** | `member-distribute` |
| **목적** | 미분배 회원을 매니저에게 배정하는 **작업(쓰기)** 페이지 |
| **개발 상태** | ✅ 구현 완료 (v3) |

**3탭 구조:**

| 탭 | 내용 | 데이터 소스 |
|----|------|-----------|
| **신규 회원 분배** | 스크리닝 결과 `routeTo=new`인 미배정 DB | `screening.service.js` |
| **중복 회원 분배** | 스크리닝 결과 `routeTo=duplicate`인 DB | 동일 번호 기존 매니저 표시 |
| **기간만료(재컨텍)** | 스크리닝 결과 `routeTo=recontact`인 DB | 과거 프로그램/미팅/결제 이력 표시 |

**검색 필터 (std-table 형식):**
- 1행: 회원검색(이름/연락처) + 성별 + 학력
- 2행: 등록일(시작~종료) + 유입경로
- 3행 (중복/재컨텍 탭): 매니저 자동완성 + 매니저 선택 모달 + 칩

**액션 버튼:**
- 매니저 드롭다운 (보유DB 수 표시) + [상담매니저 등록] + [소스외 처리]
- 재컨텍 탭: [리콜대기 등록] 추가

**분배 흐름:**
```
체크박스로 회원 선택 → 드롭다운에서 매니저 선택 → [상담매니저 등록] 클릭 → confirm → 완료
```

## 4. 스크리닝 서비스 (`screening.service.js`)

| 항목 | 내용 |
|------|------|
| **파일** | `src/services/screening.service.js` |
| **버전** | v3 (최종 확정안) |
| **개발 상태** | ✅ 구현 완료 |

### 4.1 스크리닝 함수 구조

| 함수 | 역할 | 반환 |
|------|------|------|
| `validatePhone(phone)` | 전화번호 형식 검증 (자릿수, 스팸 패턴) | `{ valid, reason }` |
| `checkDuplicate(phone, excludeId)` | 동일 번호 기존 회원 확인 (차단 아님, 분류만) | `{ duplicate, existingMember }` |
| `checkRecontact(member)` | 기간만료 재컨텍 여부 판단 | `{ isRecontact, pastHistory }` |
| `checkAutoDistribute(member)` | 자동/수동 분배 판단 (채널 기반) | `{ type: auto_roundrobin/auto_direct/manual }` |
| `screenIncomingDB(member)` | **종합 스크리닝 메인 함수** | `{ action, routeTo, reasons, ... }` |

### 4.2 스크리닝 플로우

```
1. 전화번호 형식 검증 → 형식 오류 시 action='block' (자동 소스외)
2. 기간만료(재컨텍) 체크 → routeTo='recontact' (탭3)
3. 중복 DB 체크 → routeTo='duplicate' (탭2, 차단 아님)
4. 신규 → routeTo='new' (탭1) + 자동/수동 분배 판단
```

### 4.3 자동 분배 채널 목록 (현재 구현)

## 5. 분배 방식: 자동 vs 수동

### 5.1 수동 분배 (홈페이지 유입)

- 대상: 결혼테스트 / 무료상담 / 가입비견적 / 자녀상담
- 사유: 학력·연령·매니저 성향 등 복합 판단 필요
- 처리: 팀장이 분배 페이지에서 직접 매니저 배정

### 5.2 자동 분배 (랜딩 페이지 유입)

**분배 조건 체인:**
1. **1차**: 지사 선택값 있음 → 선택 지사로 배정
2. **2차**: 지사 선택값 없음 → 지역 정보(필수값)로 지사 판정
3. **3차**: 학력 확인 → 인바운드/아웃바운드 배정

**인바운드/아웃바운드 배정:**

| 조건 | 서울본사 & 경기지사 | 부산/대구/대전/광주 |
|------|:---:|:---:|
| 대졸 이상 | → 인바운드(상담1팀) | → 인바운드(상담1팀) |
| 전문대졸 | → 아웃바운드(상담2팀) | → 인바운드(상담1팀) ⚠️ |
| 고졸 이하 | → 아웃바운드(상담2팀) | → 아웃바운드(상담2팀) |

### 5.3 매출 순위 기반 배정 순서

| 구분 | 본사 | 지방지사 |
|------|------|---------|
| **기준** | 정(순수) 매출 금액 | 전월 전체 매출 |
| **갱신 주기** | 매주 월요일 | 매월 1일 (한 달 고정) |
| **갱신 범위** | 당월 1일 ~ 직전 주 일요일 누적 | 전월 전체 |
| **순위 변동** | 매주 가능 | 없음 |
| **배정 방식** | 매출 1위부터 라운드로빈 | 매출 1위부터 라운드로빈 |

## 6. DB 상태 처리 (불가 / 변경)

### 6.1 불가 → 소스외 자동 변경

| 상태 | 의미 | 판단 기준 |
|------|------|----------|
| 불가 결번 | 전화번호 미존재 | 최소 3회 통화 시도 확인 |
| 불가 기타 | 장애인, 신용불량자 등 | 우회 안내 후 처리 |
| 불가 기혼 | 이미 결혼한 회원 | 카카오톡/통화 확인 |

### 6.2 변경 → 지사 보관 자동 환수

| 규칙 | 설명 |
|------|------|
| 이관 대상 | 변경 요청 매니저의 소속 지사로 이관 |
| 상태 복원 | "컨택전" 상태로 초기화 |
| 재배정 시점 | 신규 매니저 입사 시 |
| 핵심 | 변경 DB는 절대 삭제 금지 |

### 6.3 매니저 퇴사 시 DB 이관

- 일괄 이관 버튼 (기존 시스템 구현됨)
- **보호 DB** (이관 제외 → 후임 직접 재배정):
  - 가입 완료 (서류 미완비)
  - 가입 중 (인증 미완료)
  - 매칭 지정 미완료

## 7. 중복 DB 처리

| 상황 | 중복 여부 | 처리 |
|------|:---:|------|
| 같은 경로 + 같은 번호 2회 유입 | ✅ | 1건 삭제 (시스템 오류) |
| 다른 경로 + 같은 번호 유입 | ❌ | 정보 많은 경로의 DB를 배정 |
| 이미 배정된 회원이 재유입 | ❌ | 분배 페이지 잔류 → 수동 판단 |

> 🔒 **자동 방지**: 같은 경로 + 같은 번호 + 같은 날 = 1건만 등록

## 8. 기간만료 DB 재분배

- 기존 매니저 재직 중 → 기존 매니저에게 재배정
- 기존 매니저 퇴사 → 해당 지사 매출 순위 순차 분배
- 온라인 DB와 차이점: 주간 리셋 없이 순차 밀림 구조

## 10. As-Is vs To-Be 개선 리스트

| 구분 | As-Is (현재 시스템 한계) | To-Be (차세대 시스템 개선 요건) |
|------|------------------------|-------------------------------|
| **분배 권한 통제** | 지사 영업팀장 임의 배정, 번호 유출 리스크 | 권한 중앙화 + 대행(On/Off) + 마스킹 |
| **분배 업무 효율화** | 수백 건 수작업 분배 (업무 과부하) | 하이브리드: 홈페이지만 수동, 랜딩은 룰엔진 자동 |
| **중복 DB 처리** | 무조건 튕김/중복 혼선 | 가치 기반 필터링 (랜딩=자동삭제, 가입폼=수동 대기열) |
| **불가 DB 소스외** | 팀장 수동 확인 후 변경 | 스케줄러 자동 감지 → 소스외 (하루 2회) |
| **변경 DB 회수** | 게시판/구두 요청 → 수동 이관 | 스케줄러 자동 환수 → 컨택전 DB |
| **3개월 미컨택** | 수동 확인 후 회수 | 매월 1일 자동 회수 (보호 DB 제외) |
| **만료 정회원 귀속** | 퇴사 시 귀속 기준 모호 | 매출 1순위 릴레이 배정 + 시민권자 자동 귀속 |
| **일정/화면 동선** | 예약 시간 놓침 빈번 | UI 강제 렌더링 + Push 알림 |

## 11. 데이터 연동

### 11.1 localStorage 키 체계

| 키 패턴 | 용도 |
|---------|------|
| `purples_dist_history` | 분배 이력 (최근 50건) |
| `purples_screening_log` | 스크리닝 로그 (최근 200건) |
| `purples_status_updates` | 회원 상태 변경분 |

### 11.2 데이터 흐름

```
마케팅 채널 (웹훅) → MockAssociates 유입
        ↓
  screening.service.js (스크리닝)
        ↓
  ┌─ block  → 자동 소스외 (전화번호 형식 오류)
  ├─ new    → 탭1: 신규 분배 대기
  ├─ duplicate → 탭2: 중복 분배 대기
  └─ recontact → 탭3: 기간만료 재컨텍
        ↓
  팀장 수동 배정 or 자동 룰엔진
        ↓
  distribute-history (분배 완료 조회)
```

## 12. 개발 우선순위

### 1차 (스케줄러 자동화)
- [ ] 불가(3종) → 소스외 자동 변경 (하루 2회)
- [ ] 변경 DB → 지사 보관 + 컨택전 초기화 (하루 2회)
- [ ] 3개월 미컨택 자동 회수 (매월 1일)

### 2차 (자동분배 로직)
- [ ] 랜딩 자동분배: 지사 판정 → 학력 분류 → 매출 순위 라운드로빈
- [ ] 매출 순위 자동 산정 (본사 주간/지방 월간)
- [ ] 중복 DB 동일경로 자동 차단

### 3차 (후속 개선)
