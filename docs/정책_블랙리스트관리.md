# 블랙리스트 관리 정책

> 버전: 1.0 | 날짜: 2026-05-20 | 담당: 법무팀, 관리자

---

## 1. 개요

| 항목 | 내용 |
|------|------|
| **목적** | 서비스 부적격 회원의 재유입을 원천 차단하고, 기존 회원/매칭 상대방을 보호 |
| **적용 범위** | 준회원, 정회원, 탈회회원, 외부 유입 DB 전체 |
| **현행 문제** | 수동 기록만 가능, 재가입 원천 차단 시스템 없음 (Critical) |
| **개선 방향** | UID 기반 영구 차단 + 해시 대조 → 재유입 원천 차단 |

## 2. 블랙리스트 등록 사유 (7개 유형)

### 2.1 사유 분류표

| 코드 | 유형 | 상세 기준 | 심각도 | 등록 주체 |
|:----:|------|-----------|:------:|-----------|
| `BL-001` | **서류 위조** | 인증 서류(혼인관계, 가족관계, 소득, 건보, 졸업증명 등) 위변조 적발 | 🔴 Critical | 인증팀 → 법무팀 확인 |
| `BL-002` | **신원 허위** | 이름, 나이, 학력, 직업, 혼인상태 등 핵심 신상정보 허위 기재 | 🔴 Critical | 매니저 보고 → 법무팀 확인 |
| `BL-003` | **기혼 은닉** | 혼인관계증명서 상 기혼인데 미혼으로 가입 시도 | 🔴 Critical | 인증팀 자동 탐지 |
| `BL-004` | **성범죄/폭력** | 매칭 미팅 중 성희롱, 폭력, 스토킹 등 발생 (회원 신고 또는 경찰 접수) | 🔴 Critical | CS팀 접수 → 법무팀 |
| `BL-005` | **금전 사기** | 매칭 상대방에게 금전 요구, 사기 행위 적발 | 🔴 Critical | CS팀 접수 → 법무팀 |
| `BL-006` | **반복 클레임/악성** | 동일인 3회 이상 악의적 클레임, 직원 폭언/협박, 업무 방해 | 🟠 High | CS팀 → 관리자 판단 |
| `BL-007` | **법적 분쟁 패소** | 회사 대상 소송 후 패소 확정 또는 합의금 미이행 | 🟠 High | 법무팀 |

## 3. 등록 프로세스

```
사유 발생 (적발/신고/접수)
  │
  ├─ ① 1차 보고 ─── 발견자(매니저/인증팀/CS팀)가 사유 보고서 작성
  │
  ├─ ② 법무팀 검토 ─── 사실 관계 확인 + 증거 수집
  │                     (서류 위조: 원본 대조)
  │                     (신원 허위: 인증 서류 재검증)
  │                     (성범죄/폭력: 피해자 진술 + 경찰 접수 확인)
  │
  ├─ ③ 관리자 승인 ─── 최종 블랙리스트 등록 결재
  │
  ├─ ④ 시스템 등록 ─── 블랙리스트 DB 등록
  │                     → 전화번호 해시 등록
  │                     → 이름 + 생년월일 해시 등록
  │                     → (정회원 경험자) 주민번호 해시 등록
  │
  └─ ⑤ 후속 처리 ─── 현재 진행 중 서비스 즉시 중단
                       → 매칭 중이면 안심번호 전체 회수
                       → 프로필 URL 무효화
                       → 상태 → [블랙리스트] 전환
```

## 4. 블랙리스트 운영 2단계 구조

> ⚠️ DB 유입 시에는 **전화번호, 이름** 정도만 알 수 있으므로,
> 블랙리스트 사유 자체를 유입 시점에 탐지하는 것은 **불가능**합니다.
> 
> 블랙리스트는 **"기존 등록 건 재유입 대조"**와 **"신규 사유 탐지→등록"** 2단계로 운영합니다.

### 4.1 [1단계] DB 유입 시 — 기등록 블랙리스트 재유입 대조

이미 블랙리스트에 등록된 사람이 **번호를 바꿔서 다시 유입**하는 것을 차단합니다.

| 대조 항목 | 방식 | 가용 시점 | 비고 |
|-----------|------|-----------|------|
| **휴대폰 번호** | SHA-256 해시 비교 | DB 유입 즉시 | 가장 기본적인 대조 |
| **이름 + 생년월일** | 조합 해시 비교 | DB 유입 즉시 | 번호 변경해도 탐지 가능 |

```
신규 DB 유입
  │
  ├─ 전화번호 해시 → 블랙리스트 DB 매칭? → YES → [소스외: SRC-BLK] + 알림
  ├─ 이름+생년월일 해시 → 블랙리스트 DB 매칭? → YES → [소스외: SRC-BLK] + 알림
  └─ 미매칭 → 정상 분배 진행
```

> 💡 이 단계에서는 **새로운 블랙리스트를 발견하는 것이 아니라**,
> 과거에 등록된 블랙리스트 인원의 **재진입만 차단**합니다.

### 4.2 [2단계] 인증/서비스 단계 — 신규 블랙리스트 탐지 및 등록

실제 블랙리스트 사유가 **발견되는 시점**은 다음과 같습니다:

| 발견 시점 | 해당 사유 | 탐지 방법 | 처리 |
|-----------|-----------|-----------|------|
| **인증 단계** (정회원 전환 후) | `BL-001` 서류 위조 | API 진위확인 결과 불일치 | 가입 파기 + 블랙리스트 등록 |
| **인증 단계** | `BL-002` 신원 허위 | 서류 대조 시 학력/직업 불일치 | 가입 파기 + 블랙리스트 등록 |
| **인증 단계** | `BL-003` 기혼 은닉 | 혼인관계증명서 스크래핑 결과 | 가입 파기 + 블랙리스트 등록 |
| **인증 단계** (주민번호 입력 시) | 기등록 블랙리스트 | 주민번호 해시 대조 | **등록 차단** + 긴급 알림 |
| **매칭/서비스 중** | `BL-004` 성범죄/폭력 | 피해 회원 신고 | 서비스 중단 + 블랙리스트 등록 |
| **매칭/서비스 중** | `BL-005` 금전 사기 | 피해 회원 신고 | 서비스 중단 + 블랙리스트 등록 |
| **서비스 중** | `BL-006` 반복 악성 | CS 이력 3회 누적 | 블랙리스트 등록 |
| **탈회 후** | `BL-007` 법적 분쟁 | 소송 결과 확정 | 블랙리스트 등록 |

```
[인증 단계에서 발견]
  서류 제출 → API 진위확인 → 위조/허위/기혼 적발
  → 법무팀 검토 → 관리자 승인 → 블랙리스트 DB 등록
  → 이후 재유입 시 [1단계]에서 자동 차단

[서비스 중 발견]
  피해 회원 신고 / CS 접수 → 즉시 서비스 중단
  → 법무팀 사실 확인 → 관리자 승인 → 블랙리스트 DB 등록
  → 이후 재유입 시 [1단계]에서 자동 차단
```

### 4.3 정회원 등록 시 추가 대조 (주민번호)

정회원 등록 과정에서 주민번호가 수집되면, **추가적인 블랙리스트 대조**가 가능합니다.

```
정회원 등록 시 (주민번호 입력)
  │
  ├─ 주민번호 해시 대조 → 매칭 → 등록 즉시 차단 + 법무/관리자 긴급 알림
  │
  └─ 미매칭 → 정상 등록 진행
```

> 이름/번호를 모두 변경해도, 주민번호가 같으면 이 단계에서 잡힙니다.

---

### 4.4 요약: 언제 무엇이 가능한가

| 시점 | 할 수 있는 것 | 할 수 없는 것 |
|------|--------------|--------------|
| **DB 유입** | 전화번호/이름 해시로 **기등록** 블랙리스트 대조 | 서류 위조, 기혼 은닉 등 **신규 사유 탐지** |
| **인증 단계** | 서류 기반 위조/허위/기혼 **탐지** + 주민번호 **추가 대조** | 서비스 중 발생하는 사유 (폭력, 사기 등) |
| **서비스 중** | 피해 신고 기반 성범죄/사기/악성 **탐지** | - |

> 개인정보보호법 준수: 원본 데이터는 저장하지 않고, **해시값만 보관**하여 대조합니다.

## 5. 블랙리스트 DB 구조

| 필드 | 설명 |
|------|------|
| `id` | 블랙리스트 고유 ID |
| `phone_hash` | 전화번호 SHA-256 해시 |
| `name_birth_hash` | 이름+생년월일 조합 해시 |
| `ssn_hash` | 주민번호 해시 (있는 경우만) |
| `reason_code` | 사유 코드 (BL-001 ~ BL-007) |
| `reason_detail` | 상세 사유 텍스트 |
| `evidence_files` | 증거 파일 경로 (S3) |
| `original_member_id` | 원본 회원 ID (준회원/정회원) |
| `original_member_type` | 'associate' / 'regular' |
| `registered_by` | 등록자 (법무팀/관리자) |
| `approved_by` | 승인자 (관리자) |
| `registered_at` | 등록 일시 |
| `is_permanent` | 영구 여부 (기본 true) |
| `release_date` | 해제일 (해제된 경우) |
| `release_reason` | 해제 사유 |
| `release_by` | 해제자 |

## 6. 해제 절차

블랙리스트는 **원칙적으로 영구**이나, 아래 절차를 통해 해제 가능합니다.

| 단계 | 처리 | 담당 |
|------|------|------|
| 1 | 해제 사유서 작성 (오등록, 법적 해소 등) | 법무팀 |
| 2 | 증거 검토 (법적 분쟁 종결 증빙 등) | 법무팀 |
| 3 | 관리자 최종 승인 | 관리자 |
| 4 | 블랙리스트 DB에서 `is_permanent = false` + 해제일 기록 | 시스템 |
| 5 | [소스외] → [컨텍전] 수동 복구 가능 상태로 전환 | 관리자 |

> ⚠️ 해제 후에도 해제 이력은 영구 보존 (감사 추적용)

## 7. 블랙리스트 유형별 상세

### BL-001 서류 위조
- **탐지**: 인증팀 검증 과정에서 API 진위확인 결과 불일치
- **증거**: 위조 서류 원본 + API 검증 결과 캡처
- **처리**: 즉시 가입 파기 + 영구 차단 + 형사 고발 검토

### BL-002 신원 허위
- **범위**: 학력 허위(고졸→대졸), 직업 허위(무직→전문직), 연령 허위 등
- **탐지**: 인증 서류 대조 시 불일치 발견 또는 매칭 상대방 신고
- **처리**: 사실 확인 후 영구 차단 + 피해 회원 보상 검토

### BL-003 기혼 은닉
- **탐지**: 혼인관계증명서 스크래핑 시 혼인 이력 확인
- **자동화**: 혼인 상태 ≠ "미혼" → 자동 플래그 → 인증팀 확인
- **처리**: 확인 즉시 영구 차단

### BL-004 성범죄/폭력
- **접수**: 피해 회원 신고 또는 경찰 접수 확인
- **즉시 조치**: 안심번호 즉시 회수 + 매칭 전면 중단 + 프로필 접근 차단
- **처리**: 법무팀 사실 확인 후 영구 차단

### BL-005 금전 사기
- **범위**: 매칭 상대에게 투자 권유, 금전 차용, 보험 가입 유도 등
- **접수**: 피해 회원 신고
- **처리**: 법무팀 사실 확인 후 영구 차단 + 피해 회원 보상 검토

### BL-006 반복 클레임/악성
- **기준**: 동일인 악의적 클레임 3회 이상, 직원 폭언/협박
- **판단**: CS팀 이력 기반 → 관리자 종합 판단
- **처리**: 영구 차단 (단, 정당한 클레임은 제외)

### BL-007 법적 분쟁 패소
- **범위**: 회사 대상 손해배상 청구 후 패소, 합의금 미이행
- **처리**: 판결문 확인 후 영구 차단
