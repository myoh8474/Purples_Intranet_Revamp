**검색 필터:** 이름, 회원번호, 연락처, 상태, 프로그램, 등급, 브랜드, 담당매니저, 지사, 성별, 나이범위

#### 변경이력 UI 규칙

1. **링크 표시**: 변경이력이 있는 항목의 값은 `파란색(#2563eb) + 밑줄` 링크 스타일로 표시
2. **클릭 동작**: 값 셀 클릭 시 해당 카테고리의 변경이력 팝업(Modal) 표시
3. **팝업 구조**: 상단 변경 입력폼(해당 시) + 하단 이력 테이블
4. **데이터 소스**: `history.js` 서비스 (`getHistory`, `addHistory`) 활용
5. **이력 카테고리**: `HISTORY_CATEGORIES` 상수에 정의 (상태변경/매칭매니저/상담매니저/계약 등)

### 3.3 계약 관리 (MN-302)

**3대 계약 형태:**

| 계약 형태 | 계약 방법 | 생애주기 | 적용 프로그램 |
|----------|----------|---------|-------------|
| 인증제(1장) | 기간 | 기간 종료 → 만료 | 전문직(55만↓) |
| 인증제(2장) | 횟수/기간+기간 | 횟수or기간 종료 → 활동대기 | 전문직(100만↑), 준전문직 |
| 횟수제 | 횟수+기간 | 횟수 종료 → 활동대기 | 정규, 기타, 프로모션 |
| 기간제 | 기간+횟수 | 기간 종료 → 만료 | 정규, 기타, 프로모션 |

**핵심 규칙:**
1. 계약 시작일 = 전산 생성일 **(절대 수정 불가, 시스템 Lock)**
2. 만료일 = 룰 엔진 자동 계산 (프로그램별)
3. 1가입/2가입 통합 릴레이션 DB 관리

## 6. 매칭정보 탭 상세 기능 (MN-310)

> 매칭정보 탭은 4개 카드(메모장, 특이사항, 미팅리스트, 소개장)로 구성된 대시보드 형태이며, 각 카드에서 관리 모달을 통해 상세 작업을 수행한다.

### 6.1 소개장 관리 (소개관리 모달)

**진입**: 소개장 테이블 → 각 행의 "관리" 버튼 클릭

| 필드 | 설명 |
|------|------|
| 회원구분 | 회원당사자 / 상대회원 (자동 표시) |
| 회원명 | 소개장 데이터에서 자동 표시 |
| 소개장등록일 | **프로필 발송일로 자동 저장** |
| 답변일 | date 입력 |
| 답변선택 | 보류 / 미소개 / 의논후 결정 / 일정조율 / 소개용 |
| 비고 | 자유 입력 |

### 6.2 미팅관리 모달 (2탭 구성)

**진입**: 미팅리스트 테이블 → 각 행의 "관리" 버튼 클릭

#### 탭1: 미팅전 약속확인

회원과의 약속 조율 과정(전화/SMS/비고)을 기록·관리하는 페이지.

| 필드 | 설명 |
|------|------|
| 본인 / 상대회원 | 미팅 대상 회원 정보 자동 표시 |
| 통신유형 | ● 전화 확인 / ● SMS 발송 / ● 비고 (라디오 선택) |
| 받는이 | **[본인]** 선택 시 기본정보 번호 자동, **[기타]** 선택 시 수동 입력 |
| CallBack | **[사무실]** 02-518-9160 / **[법인폰]** / **[전국대표번호]** |
| 전송방법 | **[즉시]** / **[예약]** (예약 선택 시 datetime-local 표시) |
| 내용 | textarea 입력 |

**동작 규칙:**
1. 통신유형이 "전화 확인" 또는 "비고"인 경우 CallBack, 전송방법 행은 숨김 처리
2. 등록 시 하단 발송이력 히스토리 테이블에 행 추가 (No, 구분, 등록자, 내용, 등록일)
3. SMS 발송 시 실제 문자 발송 API 연동 (향후)

#### 탭2: 미팅등록

최종 확정된 미팅 일정을 등록하고, 안심번호 안내문자 발송을 스케줄링한다.

| 필드 | 설명 |
|------|------|
| 회원당사자 / 상대회원 | 이름 + 담당매니저 자동 표시 |
| 미팅일 | date 입력 |
| 미팅시간 | 시(0~23) / 분(0~59, 10분 단위) 분리 입력 |
| 미팅장소 | PlaceSearch 모달 연동 (장소명 + 주소 자동 입력) |
| 안심번호 | ☑ 안심번호 사용 체크 |
| 자동발송 | ☐ 자동발송 제외 체크 |

### 6.3 안심번호 안내문자 발송 프로세스

#### D-1 발송 스케줄링 규칙

| 조건 | 처리 |
|------|------|
| 미팅일 D-1이 **영업일** | D-1 당일 발송 |
| 미팅일 D-1이 **토/일/공휴일** | **직전 영업일**로 자동 이동 |
| 안심번호 사용 해제 | 일반 안내문자만 발송 |
| 자동발송 제외 체크 | 수동 발송으로 전환 (스케줄러 미등록) |

#### 공휴일 목록 (2026년 기준 내장)

신정, 설날(3일), 삼일절, 어린이날, 석가탄신일, 현충일, 광복절, 추석(3일), 개천절, 한글날, 성탄절

#### SMS 미리보기 포맷

```
[퍼플스] {상대회원명}님 안심번호:070-XXXX-XXXX / {M월 D일 (요일)} {오전/오후 N시 N분} {장소명}/ {주소} * 안내드린 안심번호가 후스콜, T전화, 단말기 자체에서 차단이 되어있는 경우에는 서비스 이용이 불가하오니 차단 유무를 확인해 주시기 바랍니다.
```

### 6.4 특이사항 등록

**진입**: 특이사항 카드 → "등록" 버튼 클릭

#### 권한별 구분 자동 결정 규칙

| 사용자 권한 (Auth.role) | 구분 값 | 선택 가능 여부 |
|------------------------|---------|:------------:|
| `admin` (시스템 관리자) | 관리/상담/매칭 전체 | ✅ 선택 가능 |
| `director` (본부장) | 관리/상담/매칭 전체 | ✅ 선택 가능 |
| `manager` (매니저) | **관리** 고정 | ❌ 자동 설정 |
| `manager` + 매칭매니저 본인 | **매칭** 고정 | ❌ 자동 설정 |
| `consultant` (컨설턴트) | **상담** 고정 | ❌ 자동 설정 |
| `legal` (법무팀) | **관리** 고정 | ❌ 자동 설정 |
| `viewer` (열람자) | - | ❌ **등록 불가** |

#### 등록 모달 구성

| 필드 | 설명 |
|------|------|
| 작성자 | `Auth.getUser().name` + 역할 자동 표시 |
| 구분 | 권한에 따라 자동 결정 또는 선택 (위 테이블 참조) |
| 내용 | textarea 입력 |

**등록 시 동작**: 특이사항 테이블 최상단에 신규 행 추가 (노란 하이라이트), 구분별 컬러코딩 유지 (관리: 파랑, 상담: 초록, 매칭: 노랑)

## 7. 상태별 상세페이지 워크플로

> 정회원의 현재 상태에 따라 상세페이지에서 활성화되는 기능 버튼과 워크플로가 달라진다.

### 7.1 상태별 활성 기능 매트릭스

| 상태 | 이벤트문자 | 계약정보확인 | 문자발송(전자서명) | 계약서조회 | 영수증조회 | 매칭기능 |
|------|:--------:|:----------:|:---------------:|:--------:|:--------:|:------:|
| **신규** | ✅ | ❌ | ❌ | ❌ | ❌ | ❌ |
| **인증중** | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ |
| **활동대기** (2가입 진행) | ❌ | ✅ | ✅ | ✅ | ✅ | ❌ |
| **활동** | ❌ | ❌ | ❌ | ✅ | ✅ | ✅ |
| **교제** (임시/교제/외부) | ❌ | ❌ | ❌ | ✅ | ✅ | ❌ (Lock) |
| **보류** (약정/임시/장기/강제) | ❌ | ❌ | ❌ | ✅ | ✅ | ❌ (Lock) |
| **탈회** (진행/완료) | ❌ | ❌ | ❌ | ✅ | ✅ | ❌ |

### 7.2 신규회원 → 이벤트 문자 발송

| 항목 | 내용 |
|------|------|
| **대상 상태** | `신규` |
| **기능** | 가입 환영 이벤트 문자 발송 |
| **트리거** | 담당 매니저가 [이벤트문자] 버튼 클릭 |
| **발송 내용** | 퍼플스 가입 환영 + 서류 안내 + 담당매니저 연락처 |

### 7.3 활동대기 → 2가입 진행 워크플로

> 1가입 만료일 도래 시 자동으로 `활동대기` 상태로 전환되며, 이 시점에 2가입 관련 기능 버튼(계약정보, 문자발송, 계약서, 영수증)이 활성화된다.

```mermaid
flowchart LR
    A["1가입 만료일 도래"] --> B["활동대기 전환\n(자동)"]
    B --> C["계약정보 확인\n(매니저)"]
    C --> D["전자서명 안내\n문자 발송"]
    D --> E["회원 전자서명\n완료"]
    E --> F["계약서/영수증\n조회 가능"]
    F --> G["인증팀\n재인증 진행"]
    G --> H["활동 전환"]
```

#### Step 1: 계약정보 확인

담당 매니저가 2가입 계약 정보를 확인하고 [확인] 버튼을 클릭한다.

**확인 항목:**

| 구분 | 필드 |
|------|------|
| 개인정보 | 이름, 아이디, 생년월일, 이메일, 전화번호 |
| 계약정보 | 계약일, 계약기간, 가입금, 성혼비, 횟수 |
| 소속정보 | 브랜드, 프로그램, 매니저, 지사 |
| 대리인 | 대리인명, 관계 |
| 주소 | 국제여부, 국가, 주소 |

- 담당 매니저가 모든 항목 확인 후 **[확인]** 버튼 클릭
- 확인 완료 시 확인 일시 + 확인자 자동 기록

#### Step 2: 전자서명 안내문 문자 발송

| 항목 | 내용 |
|------|------|
| **트리거** | [문자발송] 버튼 클릭 |
| **발송 내용** | 전자서명 URL 포함 안내문자 |
| **수신자** | 회원 기본 연락처 |
| **전제 조건** | Step 1 계약정보 확인 완료 후 활성화 |

#### Step 3: 전자서명 완료 → 계약서/영수증 조회

- 회원이 전자서명 완료 시 계약서 및 영수증 조회 가능
- **⚠ 체크사항 참조 (섹션 9)**

## 8. 회원정보 수정 권한 정책

### 8.1 수정 권한 체계

| 구분 | 수정 권한 | 프로세스 |
|------|----------|---------|
| **기본 정보** (이름, 연락처, 주소 등) | 인증팀 전용 | 매니저 → 인증팀 요청 → 인증팀 수정 |
| **계약 정보** (프로그램, 가입비, 성혼비 등) | 인증팀 전용 | 매니저 → 인증팀 요청 → 인증팀 수정 |
| **매칭 정보** (메모, 특이사항 등) | 매칭 매니저 | 직접 등록/수정 가능 |
| **상태 변경** | 각 상태별 전환 권한자 | 섹션 2 참조 |

### 8.2 수정 요청 프로세스

```mermaid
flowchart LR
    A["담당 매니저\n변경사항 발견"] --> B["인증팀에\n수정 요청"]
    B --> C["인증팀\n요청 확인"]
    C --> D["인증팀\n정보 수정"]
    D --> E["변경이력\n자동 기록"]
    E --> F["매니저에게\n완료 알림"]
```

> **핵심 원칙**: 정회원 정보의 무결성을 위해 인증팀만 직접 수정 가능하며, 모든 변경은 변경이력에 자동 기록된다.

### 8.3 2가입 완료 후 상태 전환

| 단계 | 상태 | 주체 | 조건 |
|------|------|------|------|
| 1가입 만료 | `활동` → `활동대기` | 시스템 (자동) | 만료일 도래 |
| 2가입 계약 | `활동대기` 유지 | 매니저 + 회원 | 계약정보 확인 + 전자서명 완료 |
| 재인증 | `활동대기` → `활동` | 인증팀 | 재인증 심사 완료 |
