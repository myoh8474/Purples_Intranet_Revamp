## 문제점 분석

### 1. "비즈니스 규칙" 중복 문제 ← 님 지적 동의 ✅

현재 **준회원 관리**와 **정회원 관리** 페이지 본문에 이미 운영 정책이 상세하게 작성되어 있습니다:

| 메뉴 | 본문에 작성된 정책 내용 |
|------|----------------------|
| **준회원 관리** | 회원 상태 정의(15개), 상태 변경 흐름도, 전환 정책, 개인정보 보호, 재컨택 출처 태그 |
| **정회원 관리** | 회원 상태 정의(21개), 계약 형태/규칙, 프로필 발송 규칙, 정보 수정 권한, 탈회 차단, 업그레이드/다운그레이드, 상태별 활성화 기능, 안심번호 규칙 |

> [!IMPORTANT]
> 각 메뉴의 **상단 본문에 정책이 이미 충분히 작성**되어 있으므로, 하위 페이지에 "비즈니스 규칙" 섹션을 별도로 반복 작성할 필요가 없습니다.

### 2. 기타 구조적 문제

| 문제 | 설명 |
|------|------|
| 빈 페이지 다수 | 14개 메뉴 중 11개가 비어있거나 내용 부족 |
| 공통 정책 미작성 | RBAC/DB스키마/개선로드맵 페이지가 비어있음 |
| 하위 페이지 템플릿 불일치 | 준회원 하위 페이지들은 이전 7섹션 템플릿으로 작성, 일부만 수정됨 |

## 📐 제안: 새로운 문서 구조 (5섹션)

기존 7섹션에서 **비즈니스 규칙**과 **권한 관리**를 제거하고, 정책은 상위 메뉴 본문에서 관리하는 구조입니다.

### 상위 메뉴 페이지 (준회원 관리, 정회원 관리 등) — 유지

```
[메뉴명]
├── 1. 개요 (목적, 대상 사용자, 하위 메뉴 구성)
├── 2. 운영 정책 (상태 정의, 흐름도, 규칙 등)  ← 비즈니스 규칙은 여기서 관리
├── 3. 업무 흐름 요약
└── 4. 확인 요청 사항
```

### 하위 화면 페이지 (목록, 상세, 등록 등) — 개편

```
[화면명]
├── 1. 문서 메타 (📌 버전/작성일/담당/상태)
├── 2. 화면 개요 (목적, 파일경로, 페이지ID)
├── 3. 기능 정의 (검색 필터, 테이블 컬럼, 버튼/인터랙션, 상태별 처리)
├── 4. 데이터 흐름 (데이터 소스, 읽기/쓰기 경로)
└── 5. 논의 사항 (미결정, 검토 필요 항목)
```

> [!TIP]
> **제거된 섹션:**
> - ~~5. 비즈니스 규칙~~ → 상위 메뉴 "2. 운영 정책"에서 통합 관리
> - ~~6. 권한 관리~~ → 상위 메뉴 "1. 개요 > 대상 사용자 및 권한"에서 통합 관리
>
> **이유:** 정책/권한은 화면 단위가 아니라 **메뉴 단위**로 정의하는 것이 자연스럽고, 중복 작성을 방지할 수 있음
